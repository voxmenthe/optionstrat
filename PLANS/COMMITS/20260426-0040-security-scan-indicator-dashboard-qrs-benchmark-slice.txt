Security scan dashboard: add the first benchmark-aware QRS workbench slice

This session moved the indicator dashboard into its next real architectural test: the first benchmark-aware indicator. The active plan had already pointed at `qrs_consist_excess` as the right next slice after `roc`, `roc_aggregate`, and `scl_v4_x5`, and the live checkout confirmed that choice. It is the smallest case that forces us to define how benchmark defaults, benchmark validation, date alignment, and overlap failures should behave before we touch the more coupled `scl_ma2_qrs_ma1_breakout` composite.

The main design decision was to keep the same additive seam that worked for the earlier indicators. I did not push benchmark logic into the frontend, and I did not widen the production scan contract. Instead, the new work stays inside the existing dashboard/workbench layers, while `src/backend/app/security_scan/indicators/qrs_consist_excess.py` now exposes reusable QRS-specific helpers for alignment and computation. That file gained `QrsConsistExcessAlignedInputs`, `QrsConsistExcessComputation`, `align_qrs_consist_excess_inputs()`, and `compute_qrs_consist_excess_computation()`. The existing low-level `qrs_consist_excess(...)` numeric kernel remains the math engine, but the new helper path is now the safe public seam for consumers that need dated series, resolved settings, and signal output. Production `evaluate()` delegates through the same helper path after it resolves benchmark context, so the dashboard and the scan runner are no longer two separate interpretations of QRS behavior.

The other important backend decision was to make the benchmark contract explicit at the workbench boundary before any indicator adapter runs. `src/backend/app/security_scan/indicator_adapters.py` now supports adapter-level benchmark metadata through `default_benchmark_tickers` and `required_benchmark_count`, and `src/backend/app/security_scan/indicator_workbench.py` resolves those requirements before compute. For `qrs_consist_excess`, that means exactly three unique benchmark tickers, defaulting to `SPY`, `QQQ`, and `IWM` when the request omits them, early `404` behavior when a benchmark has no usable close prices, and a clear no-common-dates failure when the stock series and benchmarks do not share any aligned dates. That is a deliberate shift away from the older implicit behavior inside `evaluate()`, where overlap losses were mostly silent and only visible if you read the indicator code.

With that contract in place, the new dashboard adapter in `src/backend/app/security_scan/indicator_adapters.py` stays narrow. It validates QRS settings through the existing strict parameter schema, uses the aligned-input helper, renders one indicator panel with `qrs`, `ma1`, `ma2`, and `ma3` traces, attaches the existing QRS signal types to either the main QRS trace or the `ma1` trace, and emits warnings when price rows are dropped because one or more benchmark closes are missing. I also added a new `IndicatorDataError` adapter-level exception so the adapter can distinguish invalid settings from unusable input data without pretending everything is a `422`.

On the frontend, the important result is what did not happen: no indicator-specific branch was needed. The existing page at `src/frontend/app/security-scan/indicators/page.tsx` already had the right shape for a benchmark-aware indicator because it was schema-driven and already carried `benchmark_tickers` in the request contract. I kept that generic path intact and only added two small improvements that help this slice without hardcoding QRS behavior. First, the benchmark input now shows a small hint when the selected indicator requires benchmark context. Second, `src/frontend/components/security-scan/IndicatorDashboardChart.tsx` now uses signal labels instead of raw signal ids in the Plotly legend, which matters because QRS signal ids are much less readable than the earlier crossover names.

The regression surface grew in the right places. `src/backend/tests/test_qrs_consist_excess.py` now locks down the new alignment helper directly. `src/backend/tests/security_scan/test_indicator_workbench.py` now covers QRS metadata, aligned multi-trace output, wrong benchmark counts, and the no-common-date failure path. `src/backend/tests/test_security_scan_api.py` mirrors that at the route layer, including the benchmark-missing `404` case. Because I changed QRS internals that are adjacent to the composite breakout indicator, I also reran `src/backend/tests/test_scl_ma2_qrs_ma1_breakout.py` as a regression check. The focused backend test set passed, and the scoped frontend TypeScript check for the touched dashboard files passed as well.

I also updated `PLANS/current-project-plans/security-scan-indicator-dashboard-plan.md` to reflect the real state after this slice. Phase 3 is now explicitly in progress rather than theoretical, the benchmark contract is documented in the shape it actually took, the QRS helper extraction is recorded as an intentional design choice, and the next detailed slice now focuses on `scl_ma2_qrs_ma1_breakout` using the public compute seams instead of repeating benchmark alignment again.

Upcoming decisions, issues, and notable findings

- The biggest remaining structural issue is that `src/backend/app/security_scan/indicators/scl_ma2_qrs_ma1_breakout.py` still owns its own benchmark/date-alignment path. That was acceptable before this slice because there was no shared QRS dashboard seam yet. It is no longer the right steady state. The next pass should make that composite consume `align_qrs_consist_excess_inputs()` and `compute_qrs_consist_excess_computation()` rather than teaching it a second alignment implementation.
- There is now a visible default-value question around QRS `map2`. The new dashboard metadata follows the indicator-module default `map2 = 21`, because that is what `qrs_consist_excess.py` actually uses when settings are omitted. The current configured scan instances in `src/backend/app/security_scan/config/scan_settings.toml` all set `map2 = 27`. We should decide intentionally whether workbench defaults are meant to represent indicator-type defaults or the most common configured scan-instance defaults.
- `src/backend/app/security_scan/indicator_adapters.py` is still cohesive after this slice, but it is no longer obviously a forever-single-file module. Once the composite breakout adapter lands, we should re-evaluate whether the adapter registry wants to split into a package so benchmark-aware and composite-specific logic do not accumulate in one file.
- Alignment-loss reporting is currently clear but intentionally lightweight. The contract exposes warnings plus `benchmark_tickers_used`, which is enough for this slice, but if the dashboard starts doing more comparative analysis we may want structured diagnostics like aligned-point counts or per-benchmark point totals instead of only human-readable warnings.
- The generic frontend contract still held. That is a positive result worth noticing. QRS added floats, benchmark requirements, multiple overlays, and multiple signal families without forcing a custom React form or chart branch. That is evidence that the metadata-driven page is still buying us real change agility.

Next best steps

1. Implement the `scl_ma2_qrs_ma1_breakout` dashboard adapter next, and make it consume the new public QRS helper path plus the existing `compute_scl_v4_x5_computation()` seam. The goal should be zero duplicate benchmark alignment logic between the composite indicator and the QRS indicator module.
2. Decide how breakout context should be visualized before writing the adapter. The main open question is whether prior highs and lows belong only in marker metadata, or whether the chart needs additional traces or lightweight diagnostics so a user can see the breakout basis directly.
3. Reuse the current frontend page unchanged if possible. If the breakout slice needs more UX, prefer adding metadata-driven or diagnostics-driven hints rather than branching on indicator id.
4. After the composite slice lands, reassess the shape of `indicator_adapters.py`. If the new adapter materially increases mixed concerns, split at that point rather than preemptively.
5. Run a live browser smoke pass with both dev servers up once the breakout slice is in place. The contract and types are in good shape, but the next unanswered questions are visual: whether the QRS legend remains readable with multiple signal families, whether the benchmark-warning text is prominent enough, and how a composite breakout should look without cluttering the lower panel.
6. Resolve the QRS default-value question explicitly. If `map2 = 27` is the operational default the team actually wants everywhere, change the indicator module and dashboard metadata together in a dedicated pass. If `21` is the intentional indicator-type default and `27` is only a scan-instance choice, leave the code as-is and document that distinction more explicitly.
