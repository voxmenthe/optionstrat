Security scan dashboard: add `roc_aggregate` workbench support and shared compute path

This session moved the indicator workbench from its first single-trace proof point into the first real multi-trace case. The plan already pointed at `roc_aggregate` as the right next slice, and the current repo state confirmed that choice. It exercises the two important extensions we needed next without pulling in benchmark complexity: multiple indicator traces on one panel, and schema-driven `integer_list` settings. That made it the right pressure test for the dashboard contract before touching `scl_v4_x5` or any QRS-based indicator.

The most important design decision in this slice was to avoid growing the dashboard adapter into another client of private indicator internals. `src/backend/app/security_scan/indicator_adapters.py` now has a new `roc_aggregate` dashboard adapter with metadata for `roc_lookbacks`, `roc_change_lookbacks`, `ma_short`, and `ma_long`, plus a compute path that returns three traces on the indicator panel: the aggregate score, the short moving average, and the long moving average. The adapter also emits dashboard signal markers for `cross_above_both` and `cross_below_both`, anchored to the aggregate score trace, and keeps the contract strict by validating dashboard settings through `resolve_adapter_settings()` before any indicator math runs. That means the dashboard requires real JSON arrays for list settings, rejects unknown keys, and keeps the request shape explicit and predictable.

Instead of teaching the adapter how `roc_aggregate` works internally, I pulled the reusable indicator math into a small public pure-compute function inside `src/backend/app/security_scan/indicators/roc_aggregate.py`. The new `compute_roc_aggregate_computation()` result packages resolved settings, usable close series, score series, short and long moving-average series, and the production crossover signals. The existing production `evaluate()` function now delegates to that compute result instead of recomputing the same graph separately. This was the lowest-friction path once I saw two things in the repo: the dashboard needed full series, and notebook tooling in `src/backend/notebooks/indicator_sanity_check.py` was already reaching into `roc_aggregate` private helpers. The new public compute helper keeps the indicator math in the indicator module, keeps the dashboard adapter focused on visualization assembly, and reduces the chance that production scan behavior and dashboard behavior drift apart over time.

On the frontend, the good news was that the page really was generic enough for this second case. `src/frontend/app/security-scan/indicators/page.tsx` already builds its controls from backend metadata and serializes opaque `settings`, so `roc_aggregate` dropped into the existing flow without any indicator-specific React logic. I only made two small generic improvements that became worthwhile once list parameters and multi-trace diagnostics were real rather than hypothetical. First, the page now shows `parameter.description` under each generated field, which makes comma-separated list expectations visible instead of implicit. Second, dashboard warnings now render as a visible amber strip rather than being buried in the diagnostics JSON. That matters for the insufficient-history case, where an empty chart is expected behavior and should be explained in the UI rather than treated like a silent failure.

The backend contract coverage grew in the right places. `src/backend/tests/security_scan/test_indicator_workbench.py` now asserts stable metadata order for both `roc` and `roc_aggregate`, checks the `roc_aggregate` parameter schema, verifies the three-trace compute payload on synthetic prices, verifies the expected cross-above and cross-below signal dates, and locks in the strict list-setting behavior by rejecting scalar `roc_lookbacks` in the dashboard contract. It also covers the important warm-up path where there are usable prices but not enough history to compute the aggregate score yet; that returns a successful response with empty traces and a warning, rather than pretending it is a hard no-data failure. `src/backend/tests/test_security_scan_api.py` now mirrors that coverage at the route layer, including a successful `roc_aggregate` compute response and a `422` case for bad list settings.

Because I refactored the production indicator module rather than only the dashboard adapter, I also reran the pre-existing `roc_aggregate` production tests in `src/backend/tests/test_security_scan.py`. Those still pass, which is the key safety check for this session. The workbench additions are additive, but the shared compute extraction does touch a load-bearing indicator path, so it needed that extra confirmation.

The validation that actually ran in this session was deliberately narrow and relevant. `uv run pytest src/backend/tests/security_scan/test_indicator_workbench.py src/backend/tests/test_security_scan_api.py` passed. `uv run pytest src/backend/tests/test_security_scan.py -k roc_aggregate` passed, confirming the production `evaluate()` behavior still matches the existing scan expectations. The scoped dashboard frontend type check also passed, using the local frontend compiler directly from `src/frontend` via `src/frontend/node_modules/.bin/tsc --noEmit ...`. That detail mattered because invoking `npx tsc` from the repo root pulled the npm placeholder package rather than the frontend toolchain, which is an environment quirk rather than a dashboard defect.

The project plan in `PLANS/current-project-plans/security-scan-indicator-dashboard-plan.md` was updated to reflect the new real state. Phase 2 now records `roc_aggregate` as completed, documents the shared pure-compute decision, notes the stricter dashboard list-setting contract, records the generic frontend warning/help improvements, and expands the next detailed slice around `scl_v4_x5` so the next implementation pass can start from a concrete shape rather than another open-ended exploration.

Upcoming decisions, issues, and notable findings

- `scl_v4_x5` is now the right next indicator, but it should not be added by teaching the adapter a pile of indicator-specific internals. If the adapter would need several underscore helpers or duplicate the countdown math, it deserves the same treatment `roc_aggregate` just got: expose one public pure-compute result in `src/backend/app/security_scan/indicators/scl_v4_x5.py` and keep visualization assembly separate.
- The dashboard contract for `integer_list` settings is intentionally stricter than the production `roc_aggregate` helper path. In the dashboard, `roc_lookbacks` and `roc_change_lookbacks` must be JSON arrays. In the production indicator internals, a scalar integer is still tolerated and coerced to a one-item list. That mismatch is now explicit. It is defensible, but it is a real contract decision worth either preserving intentionally or flattening later for consistency.
- Notebook tooling is still a pressure point. `src/backend/notebooks/indicator_sanity_check.py` continues to use `roc_aggregate` private helpers directly. The new public compute function makes that technical debt visible. If we keep adding public compute helpers for dashboard reuse, the notebooks should probably migrate onto them instead of maintaining a parallel private-helper integration path.
- Full-project frontend validation is still not clean for reasons unrelated to this slice. `src/frontend/app/options/page.tsx` still blocks `npm run build` because `OptionContract.last` is referenced on a type that does not define it. That issue predates this session, but it remains a real confidence drag every time a frontend change lands.
- Linting is still not a reliable gate. The frontend has ESLint 9 but no `eslint.config.(js|mjs|cjs)` file, so direct lint execution is still blocked by tooling configuration rather than by dashboard code quality.
- Running backend tests still modifies tracked runtime artifacts such as `.coverage` and `src/backend/logs/backend.log`. That is not new, but it remains worth deciding whether those files belong under version control at all, because they keep showing up as noise after normal validation work.

Next best steps

1. Add `scl_v4_x5` as the third dashboard indicator, and treat it as the real structural test of the adapter model. Define the exact chart contract first: primary countdown trace, `MA1`, `MA2`, zero/reference line, and seven-bar high/low markers.
2. Before wiring that adapter, decide whether `src/backend/app/security_scan/indicators/scl_v4_x5.py` should expose a public pure-compute function the way `roc_aggregate` now does. If the adapter would otherwise need more than one or two internal helpers, the answer should be yes.
3. Add focused synthetic OHLC tests for the first `scl_v4_x5` dashboard payload. The important contracts are not just successful output, but also how missing `high` or `low` values fail or warn, and which trace key the signal markers should target.
4. Run a live browser smoke pass for the current `roc_aggregate` workbench route once the next session has a backend and frontend dev server up. The current automated checks prove the contract and types, but they do not yet verify legend readability, marker clutter, or warm-up-warning clarity in the actual Plotly UI.
5. Decide whether to keep the dashboard list-setting contract stricter than production. If consistency matters more than strict request shape, adjust one side intentionally. If strictness is the goal, document it clearly and consider migrating the notebook helpers away from their more permissive private parsing path.
6. Clear the unrelated frontend `OptionContract.last` build blocker and the ESLint 9 config gap before too many more dashboard UI changes land. Those are not dashboard features, but they are now the main obstacles to stronger whole-frontend validation after each slice.
