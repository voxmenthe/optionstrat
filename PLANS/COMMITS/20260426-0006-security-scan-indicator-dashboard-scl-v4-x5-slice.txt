Security scan dashboard: add `scl_v4_x5` workbench support and preserve the production SCL seam

This session completed the next planned dashboard slice by adding `scl_v4_x5` as the third supported workbench indicator. That sounds like a straightforward adapter addition, but the real work was deciding where the logic should live so we did not repeat the mistake the plan was already warning about: teaching the dashboard layer too much about indicator internals.

The current repo state made that decision fairly clear. `roc_aggregate` had already established the right seam in `src/backend/app/security_scan/indicators/roc_aggregate.py`: expose one public pure-compute helper that resolves settings once, returns dated series, and lets both production `evaluate()` and dashboard visualization assemble from the same source of truth. `scl_v4_x5` needed the same treatment. The dashboard wanted full `CountdownDisplay`, `MA1`, and `MA2` series plus seven-bar markers, but the existing public wrapper only exposed countdown points and `evaluate()` only returned signals. At the same time, `src/backend/app/security_scan/indicators/scl_ma2_qrs_ma1_breakout.py` already depends directly on the raw `scl_v4_x5()` dict contract, and its tests monkeypatch that exact function and key shape. That meant the lowest-blast-radius choice was additive extraction, not contract replacement.

`src/backend/app/security_scan/indicators/scl_v4_x5.py` now exposes `compute_scl_v4_x5_computation()`. The new helper resolves the nine integer settings once, extracts usable OHLC rows, returns dated countdown, `MA1`, and `MA2` series, carries the indicator’s own seven-bar high and low signals, and tracks how many price rows were skipped because `high` or `low` were missing. The existing raw `scl_v4_x5()` function is still there, still returns the same dict keys, and still serves the composite breakout indicator without forcing that caller to move at the same time. `compute_countdown_series()` and `evaluate()` now delegate through the shared helper, which means the dashboard and the production scan logic are no longer two separate interpretations of the same indicator math.

That helper is what made the dashboard adapter stay simple. `src/backend/app/security_scan/indicator_adapters.py` now registers `scl_v4_x5` with a schema of nine integer parameters: `lag1` through `lag5`, `cd_offset1`, `cd_offset2`, `ma_period1`, and `ma_period2`. The adapter renders one lower panel with `countdown`, `ma1`, and `ma2` traces plus a zero reference line. It intentionally maps signal markers from the indicator’s own seven-bar logic instead of the scan runner’s separate five-bar summary flags. That distinction matters. The scan runner uses `compute_prior_window_flags()` for a different summary purpose in `scan_runner.py`; the dashboard needed to stay faithful to `scl_v4_x5` as an indicator rather than quietly inherit a neighboring summary convention.

The other important backend decision was diagnostics behavior for OHLC-sensitive indicators. The workbench already shows close-price points in the top panel, but `scl_v4_x5` needs `high` and `low` as well. Rather than turning partial OHLC rows into a hard failure, the adapter now emits a warning when rows are skipped for missing `high` or `low` fields. That keeps the chart honest without pretending the underlying data problem does not exist. I kept `diagnostics.price_points` aligned with the top-panel close series instead of redefining it to mean “usable OHLC rows,” because the dashboard header should still describe what the user actually sees in the price panel.

On the frontend, the main result is that nothing indicator-specific had to be added to the form layer. That is a positive result, not missing work. `src/frontend/app/security-scan/indicators/page.tsx` and `src/frontend/lib/security-scan/indicatorParameterInputs.ts` already handled nine integer inputs through the existing schema-driven contract, which is good evidence that the metadata shape is still carrying its weight. The only frontend code change was in `src/frontend/components/security-scan/IndicatorDashboardChart.tsx`, where signal marker styling now recognizes `high` and `low` semantics in addition to the earlier `up` and `down` crossover naming. Without that generic tweak, `seven_bar_high` and `seven_bar_low` would have rendered as the default blue circles, which would have been technically functional but semantically weak.

The backend regression surface grew in the right places. `src/backend/tests/security_scan/test_indicator_workbench.py` now expects `scl_v4_x5` in metadata, verifies the nine-parameter schema, checks the three-trace dashboard payload on synthetic OHLC data, locks in the seven-bar-high signal dates, covers warnings when OHLC rows are partially missing, and rejects invalid integer settings such as `lag1 = 0`. `src/backend/tests/test_security_scan_api.py` mirrors that at the route layer so the HTTP contract is pinned as well. Because I touched a production indicator module rather than only dashboard glue, I also added a new focused unit test file at `src/backend/tests/test_scl_v4_x5.py` for the computation helper itself. And because `scl_ma2_qrs_ma1_breakout` already depends on the raw SCL function, I reran `src/backend/tests/test_scl_ma2_qrs_ma1_breakout.py` as a regression check rather than assuming the additive extraction was harmless.

The validations that actually ran were tight and relevant. `uv run pytest tests/test_scl_v4_x5.py tests/test_scl_ma2_qrs_ma1_breakout.py tests/security_scan/test_indicator_workbench.py tests/test_security_scan_api.py` passed from `src/backend`. The scoped dashboard TypeScript check also passed using the local frontend toolchain from `src/frontend` via `node_modules/.bin/tsc --noEmit ...`. I did not run live backend or browser smoke calls in this session, so the visual confirmation of the new legend and marker behavior in a running Plotly page is still pending. The unrelated full frontend build blocker is still unchanged: `src/frontend/app/options/page.tsx` references `OptionContract.last` on a type that does not define it. Linting is still blocked by the existing ESLint 9 configuration gap rather than by dashboard code.

The active plan in `PLANS/current-project-plans/security-scan-indicator-dashboard-plan.md` was updated to close out Phase 2 as completed. More importantly, the plan now records why `scl_v4_x5` was added through a public compute helper instead of a broader refactor, why the raw function contract was preserved for existing callers, how missing-OHLC warnings behave, and what the next benchmark-aware slice should do before we touch composite indicators.

Upcoming decisions, issues, and notable findings

The next real decision is no longer about frontend form mechanics. The schema-driven page has now survived `roc`, `roc_aggregate`, and `scl_v4_x5`, including a nine-parameter integer-only indicator, without needing indicator-specific React code. That means the real pressure has moved to the backend benchmark boundary. Phase 3 should begin with `qrs_consist_excess`, not `scl_ma2_qrs_ma1_breakout`, because the first unsolved structural problem is benchmark alignment, missing benchmark failure behavior, and date-set intersection. If we skip that and jump straight to the composite indicator, we will be solving two hard problems at once and the code will start fighting back.

There is also a now-visible cohesion question around `src/backend/app/security_scan/indicator_adapters.py`. The single-file approach was the right call while the contract was still earning its shape. After `scl_v4_x5`, we now have three concrete adapters and one of them is OHLC-sensitive. A fourth adapter with benchmark alignment logic may be the point where splitting into an adapter package actually reduces change blast radius rather than increasing it. I would not split it preemptively before the first QRS slice lands, but I would now treat that split as a live architectural question rather than a distant cleanup idea.

The scan-runner relationship is also worth watching closely. `scan_runner.py` uses `compute_countdown_series()` and `compute_prior_window_flags()` for five-bar summary fields, while the dashboard intentionally uses the indicator’s own seven-bar signals from `evaluate()`. That separation is correct, but it means future contributors could easily conflate “SCL summary flags” with “SCL indicator signals” unless we keep being explicit about which contract is being visualized where.

Another decision that still deserves attention is the consistency boundary between production indicator helpers and the dashboard contract. `roc_aggregate` remains more permissive internally than the dashboard request schema, because the dashboard insists on real JSON arrays for list parameters. `scl_v4_x5` does not have that same list-setting issue, but the broader question remains: do we want the dashboard to be the stricter typed boundary and let internals stay more permissive, or do we eventually want to flatten those differences so helper behavior and API behavior align exactly? There is no urgent defect here, but the answer affects how much translation logic accumulates over time.

The final notable finding is operational, not architectural. Normal targeted backend validation still dirties tracked runtime artifacts such as `src/backend/.coverage` and `src/backend/logs/backend.log`, and that noise remains in the repo after a clean test run. The plan already calls this out, and it is still a real productivity drag because it blurs the difference between intentional code changes and routine local execution residue.

Next best steps

1. Start Phase 3 with a public pure-compute helper for `qrs_consist_excess` if the first dashboard adapter would otherwise need multiple private helpers or duplicated alignment logic. Follow the same rule that worked for `roc_aggregate` and `scl_v4_x5`: indicator math stays in the indicator module, visualization assembly stays in the adapter.

2. Before implementing that adapter, lock down the benchmark alignment contract in `indicator_workbench.py`. Decide exactly how we intersect ticker dates with benchmark dates, whether any partial output is acceptable, and what a missing-benchmark failure looks like at the API boundary. This should be explicit before the first line of adapter code, because hidden alignment loss is the highest-risk source of misleading charts in the next phase.

3. Add `qrs_consist_excess` as the first benchmark-aware dashboard payload with one panel containing the main QRS trace plus `MA1`, `MA2`, and `MA3`, then map the existing QRS signals onto trace-targeted markers. Keep that slice singular. Do not combine it with composite breakout logic yet.

4. After `qrs_consist_excess` lands, re-evaluate `indicator_adapters.py` for a split into an adapter package. If the QRS slice adds enough benchmark-specific glue that the file starts mixing unrelated concerns, that will be the right time to divide by adapter or by helper family.

5. Run a live local smoke pass for `scl_v4_x5` in the actual page once the next session has both dev servers available. The contract is tested, but the human questions still need a browser: whether the legend remains readable with three traces, whether the seven-bar markers are visually intuitive, and whether the skipped-OHLC warning is obvious enough when the top and bottom panels use different subsets of rows.

6. Keep the unrelated frontend validation blockers in view. The `OptionContract.last` type problem and the missing ESLint 9 flat config are outside this dashboard slice, but they are still what prevent stronger whole-frontend validation from becoming routine after each dashboard change. If those remain unresolved for too long, each subsequent frontend slice will carry more uncertainty than it should.
